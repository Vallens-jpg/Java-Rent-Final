2. Logical Record Structure (LRS)

LRS merepresentasikan struktur tabel fisik pada database relasional (MySQL). Panah menunjukkan relasi dari Foreign Key (FK) ke Primary Key (PK).

classDiagram direction LR class users { +id : unsignedBigInteger (PK) name : varchar email : varchar password : varchar phone : varchar nik : varchar address : text ktp_photo : varchar role : enum('admin', 'customer') created_at : timestamp updated_at : timestamp } class cars { +id : unsignedBigInteger (PK) brand : varchar size : varchar transmission : varchar plate_number : varchar price_per_hour : decimal(10,2) status : enum('available', 'rented', 'maintenance') image : varchar created_at : timestamp updated_at : timestamp } class rentals { +id : unsignedBigInteger (PK) #user_id : unsignedBigInteger (FK) #car_id : unsignedBigInteger (FK) start_date : datetime end_date : datetime total_price : decimal(10,2) status : enum('pending', 'active', 'completed', 'cancelled') created_at : timestamp updated_at : timestamp } class penalties { +id : unsignedBigInteger (PK) #rental_id : unsignedBigInteger (FK) amount : decimal(10,2) reason : varchar status : enum('unpaid', 'paid') created_at : timestamp updated_at : timestamp } %% LRS Relationships (Foreign Key Mappings) rentals "FK: user_id" --> "PK: id" users rentals "FK: car_id" --> "PK: id" cars penalties "FK: rental_id" --> "PK: id" rentals

3. Kamus Data & Penjelasan Entitas

Penjelasan detail mengenai fungsi setiap entitas (tabel), atribut (kolom) di dalamnya, serta alasan logis mengapa data tersebut wajib ada dalam sistem arsitektur Drivora.

Tabel users

Mengelola data otentikasi (login) dan profil seluruh pengguna sistem.

Mengapa harus ada? Tanpa tabel ini, sistem tidak akan tahu siapa yang meminjam mobil dan siapa yang bertugas sebagai Admin untuk menyetujui pesanan.

  • id (PK) : Kunci utama unik (*BigInteger*) untuk mengidentifikasi setiap *user* secara spesifik.
  • name & email : Informasi dasar untuk komunikasi dan *username* saat login.
  • password : Kata sandi yang telah dienkripsi (Hash) untuk keamanan akun.
  • phone : Nomor telepon, mutlak diperlukan agar Admin/Supir dapat menghubungi *customer* saat serah terima unit.
  • nik & ktp_photo : Identitas kependudukan untuk validasi keamanan dan jaminan sewa (Diunggah oleh Admin).
  • address : Alamat tempat tinggal untuk validasi domisili.
  • role : Menentukan otorisasi (*Enum: admin/customer*). Sangat krusial agar *customer* tidak bisa mengakses *dashboard* Java milik Admin.

Tabel cars

Master Data (Katalog) aset mobil yang dimiliki oleh perusahaan Drivora.

Mengapa harus ada? Ini adalah "nyawa" atau inventaris bisnis rental. Tabel ini menjadi pusat pencarian (*Single Source of Truth*) ketersediaan barang.

  • id (PK) : ID unik setiap unit mobil.
  • brand & image : Menampilkan merek (cth: Toyota Avanza) dan URL foto untuk kepentingan *marketing* di *frontend*.
  • size & trans... : Spesifikasi (Kapasitas Seat & Transmisi) agar *customer* bisa melakukan *filter* pencarian sesuai kebutuhan mereka.
  • plate_number : Nomor polisi (Unik). Sangat penting untuk identifikasi fisik di dunia nyata saat tilang atau masalah asuransi.
  • price_per_hour: Tarif sewa untuk mengalkulasi total tagihan secara otomatis (*auto-calculation*).
  • status : *Enum* (*available, rented, maintenance*). Wajib ada agar mobil yang sedang disewa tidak bisa di-*booking* ganda oleh *customer* lain.

Tabel rentals

Tabel Transaksional untuk mencatat sejarah dan status peminjaman.

Mengapa harus ada? Menghubungkan "Siapa (User) meminjam Apa (Car)". Tanpa tabel ini, perusahaan tidak memiliki bukti histori pemesanan dan arus kas pendapatan.

  • id (PK) : Nomor order pesanan.
  • user_id (FK) : Menyambung ke tabel `users` untuk mengetahui data peminjam.
  • car_id (FK) : Menyambung ke tabel `cars` untuk mengunci unit yang dipesan.
  • start & end date: Durasi peminjaman. Krusial untuk menghitung total biaya dan mengetahui kapan mobil harus dikembalikan.
  • total_price : Harga akhir yang disetujui (Durasi × `price_per_hour`).
  • status : Jejak pergerakan transaksi (*pending* -> *active* -> *completed*).

Tabel penalties

Mengelola denda keterlambatan atau kerusakan terkait suatu pesanan.

Mengapa harus ada? Untuk penegakan aturan (*policy enforcement*). Pisahkan dari tabel `rentals` agar satu penyewaan bisa memiliki beberapa catatan denda jika pelanggaran terjadi berlapis.

  • rental_id (FK) : Menghubungkan denda ke nomor pemesanan yang mana.
  • amount : Besaran denda (Rupiah) yang harus dibayar.
  • reason : Alasan denda (misal: "Telat pengembalian 3 Jam"). Penting untuk transparansi agar *customer* tidak komplain.
  • status : *unpaid* / *paid*. Agar sistem terus menagih peringatan di *dashboard* sampai denda dilunasi.
